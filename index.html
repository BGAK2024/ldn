Introduction

Linked Data Notifications is a specialised use of Linked Data Platform for sending and consuming notifications. It is not dependent on a complete implementation of LDP but designed to be compatible. This specification describes the particular features necessary to make it easy to exchange notifications in a decentralised, interoperable way.

This protocol assumes a modular approach to creating and storing data, in particular a decoupling between applications and data storage. Thus, where senders, receivers and consumers of notifications are independently implemented and run on different technology stacks, this protocol allows them to seamlessly work together.

Overview

A sender is triggered, either by a human or an automatic process, to deliver a notification to a server. The notification is data intended for the attention of the receiver, for example: a personal message from a friend; a pingback link; a comment on a blog post; an invitation to collaborate; a calendar reminder.

The sender discovers where to deliver the notification based on the addressee or subject of the notification - the receiver - who advertises the appropriate location to send it to. The receiver also exposes the notification data (according to appropriate access control) for use by consumers.

Consumers discover the notifications in the same way as the sender, and may perform further processing on it, combine it with some other data, or simply present it in a suitable human-readable way.

Protocol

This section describes the discovery of the URL to which notifications may be delivered or read from (the Inbox) and the delivery mechanism. Notification contents are described in Payload.

Sending

Senders MUST support delivering notifications through both POST and PATCH.

Senders SHOULD first make an OPTIONS request to the Inbox; if Accept-Post header is returned the sender MUST use POST, and if the Accept-Patch header is returned the sender MUST use PATCH.

Senders MAY make a HEAD request on the Inbox to determine how to send the notification. If the Inbox returns an HTTP Link header [RFC5988] with rel="type" and the target contains http://www.w3.org/ns/ldp#Container the sender MUST use POST. If the target contains http://www.w3.org/ns/ldp#RDFSource (and does not contain http://www.w3.org/ns/ldp#Container) the sender MUST use PATCH.

If there is no adequate response from the OPTIONS or HEAD requests, or the sender cannot make either request, the sender MUST try POST first, and if that fails with HTTP 405 Method Not Allowed or HTTP 406 Not Acceptable, try PATCH.

The sender MAY use a HEAD request to determine the RDF content types accepted by the server, and serialize the notification in the request body according to the value of the Accept header returned. Otherwise, the body of the POST or PATCH request MUST contain the notification in JSON-LD.

Receiving

Receivers MUST advertise an Inbox URL. In LDP terms, an Inbox may be either a Container or a Resource. Receivers MUST support GET requests on the Inbox. Receivers MUST also support either POST (for Container) or PATCH (for Resource) at this URL (not both).

A GET request on the Inbox MUST eventually return the notifications in the Inbox with HTTP 200 OK using the applicable RDF serialisation. In LDP terms, if the Inbox URL is a Container, it MUST return a description with either membership or containment triples. If the Inbox URL is a Resource, it MUST return the complete resource.

To support POST: OPTIONS or HEAD requests on the Inbox SHOULD return an Accept-Post header and MAY return Link: <http://www.w3.org/ns/ldp#Container>; rel="type". Upon receipt of a POST request to the Inbox, the receiver MUST create a new resource containing the RDF data from the body of the POST request and MUST return either HTTP 200 OK, HTTP 201 Created, HTTP 202 Accepted or HTTP 204 No Content. In the case of a 200 a Content-Location header pointing to the created resource MUST also be returned. In the case of a 201 a Location header pointing to the created resource MUST also be returned. In the case of a 204 a Content-Location header pointing to the created resource SHOULD also be returned.

To support PATCH: OPTIONS or HEAD requests on the Inbox SHOULD return an Accept-Patch header and MAY return Link: <http://www.w3.org/ns/ldp#RDFSource>; rel="type", and MUST NOT return Link: <http://www.w3.org/ns/ldp#Container>; rel="type". Upon receipt of a PATCH request to this URL, the receiver MUST update the resource at the URL to append the data from the body of the PATCH request and MUST return either HTTP 200 OK, HTTP 202 Accepted or HTTP 204 No Content.

To support OPTIONS: servers MUST enable CORS pre-flight requests [cors].

Receivers MUST accept notifications where the request body is JSON-LD, with the Content-Type: application/ld+json, and MAY accept other RDF content types (eg. text/turtle, text/html). Receivers SHOULD advertise the content types they accept with an Accept header on the Inbox URL. Receivers MAY reserialize a notification into a different syntax upon delivery, prior to doing anything else with it.

If a request is made to the Inbox with an HTTP method the receiver does not support, a HTTP 405 Method Not Allowed error code MUST be returned. If the request is made with a content type the receiver does not support, a HTTP 406 Not Acceptable error code MUST be returned.

Consuming

Inbox URLs may enter consuming systems either by direct input or through discovery.

Additional fetching or inferring additional information from the payload is at the discretion of the consumer.

Discovery

Senders and consumers MUST either fetch the target URL (and follow redirects) or make a HEAD request and check for an HTTP Link header with a rel value of http://www.w3.org/ns/ldp#inbox. If no such Link header found, consumers MUST use the object URI(s) of the predicate http://www.w3.org/ns/ldp#inbox in RDF Sources [LDP-RS] (e.g., serialised as Turtle, JSON-LD, or HTML+RDFa). Consumers MAY utilise application specific relations (e.g., sioc:has_discussion, as:replies, oa:annotationService) to discover specific Inbox URLs for any subject URI provided that the Inbox URL is an instance of http://www.w3.org/ns/ldp#Inbox. As a last resort, consumers MAY want to look for an HTML element with a rel containing value http://www.w3.org/ns/ldp#inbox along with the corresponding attribute for the destination URL.

Payload

To allow for a wide variety of use cases, the actual payload of the notification (the request body) is deliberately left to the discretion of the sender. The sender MUST NOT assume that the receiver can fetch or infer anything additional from the payload, and thus MUST include everything they want the receiver to know.

Authorization

The owner of the inbox typically sets its ACL rules and thereby for the notifications it contains. By default, when profile inboxes are created, inboxes must be public append only as they are inherently private. Other types of inboxes e.g., for blog posts, may be set to public read and append so that applications can use a follow-your-nose approach to discover and display the contents of a notification, as well as submit new notifications. Further, the ACL settings for the inbox may determine which agents can send notifications.

Integrity

Constraints: Inbox URLs SHOULD announce their own shape constraints (e.g., SHACL, Web Annotation Protocol) via an HTTP Link header with a rel value of http://www.w3.org/ns/ldp#constrainedBy at a target URL. Senders SHOULD comply with constraint specifications.
